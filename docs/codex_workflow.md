# Codex Workflow Playbook

这份文档的目标不是教你“怎么提问更礼貌”，而是把 Codex 的工作负载压缩到最有价值的部分：少铺背景、少扫无关目录、少在一个对话里混多个目标。

## 1. 你当前最容易浪费 token 的地方

- 一个对话里同时做“改代码 + 查原因 + 设计方案 + 跑大构建”
- 每次都从工作区根目录重新解释整个项目
- 把大段日志、整文件源码、多个模块背景一次性塞进上下文
- 明明只改一个包，却让模型做全仓搜索和全仓构建
- 任务切换后继续沿用旧线程，导致旧约束和新目标互相污染

## 2. 核心原则

### 2.1 一次对话只做一个目标

推荐粒度：

- 一个 bug
- 一个 feature
- 一个 review 范围
- 一个重构切片

不推荐：

- “顺便”
- “再看看”
- “一起把这些都改了”

如果目标变了，直接开新对话。不要试图在长线程里“重置上下文”。

### 2.2 先给边界，再给内容

先说：

- 目标
- 范围
- 约束
- 验证方式

再补：

- 现象
- 猜测
- 日志摘录

这样模型会先建立任务边界，而不是先吸收大量背景。

### 2.3 永远只给最小必要上下文

优先级应该是：

1. 包名
2. 文件路径
3. 报错签名
4. 复现命令
5. 关键日志片段

最后才是长篇背景说明。

### 2.4 验证也要收敛

优先：

- 单包构建
- 单测试用例
- 单 launch
- 单复现步骤

避免默认全仓 `colcon build`，除非你确实改了跨包接口。

## 3. 推荐的人机协作协议

每次发任务，尽量使用下面这个短结构：

```md
目标: 修复 `sp_vision25` 中 xxx 问题
范围: 只允许改 `src/sp_vision25`
约束: 不改消息定义，不引入新依赖
验证: `colcon build --packages-select sp_vision25`
现象: 启动后 xxx，日志里有 yyy
```

这个结构的好处：

- Codex 会先缩小搜索范围
- 它不需要猜你能接受什么改动
- 它知道该跑什么验证，不会默认跑大构建

更完整的规范和教程见 [docs/codex_usage_tutorial.md](/home/ats/ats_sentry_ws/docs/codex_usage_tutorial.md)。

## 4. 不同任务的最优打开方式

### 4.1 修 bug

你发：

```md
目标: 修复 xxx
范围: `src/sp_vision25`
复现: `ros2 run ...`
期望: 不再出现 xxx
验证: `colcon build --packages-select sp_vision25`
```

不要一上来就贴 300 行日志。先给错误签名，必要时再补片段。

### 4.2 做 feature

你发：

```md
目标: 增加 xxx 能力
范围: `src/pb2025_sentry_behavior`
约束: 保持现有 topic 和参数兼容
验证: 编译通过，并说明入口文件和行为变化
```

重点是兼容性边界和验收条件，不是把设计史全讲一遍。

### 4.3 做 review

你发：

```md
请 review 最近改动，只关注 bug、回归风险、缺失验证。
范围: `src/sp_vision25`
验证: 不需要改代码，先给 findings
```

这会强制模型进入 review 模式，而不是默认开始改。

## 5. 什么时候必须开新对话

出现下面任一情况，直接开新线程：

- 模块变了，比如从视觉切到导航
- 目标变了，比如从修 bug 变成做方案
- 约束变了，比如从“只改一个包”变成“允许跨包”
- 你已经连续追问 6 到 10 轮，而且开始反复解释前文
- 线程里已经混入大量旧日志、旧方案、旧假设

新线程的开场不要重写全部背景，只要贴 handoff 摘要。

## 6. 推荐 handoff 机制

每轮任务结束，让 Codex 给你一个 5 行以内 handoff：

- 做了什么
- 改了哪些路径
- 还有什么没验证
- 下一轮该从哪里继续

下一次开新对话时，直接粘这个 handoff，而不是粘整个历史。

## 7. 你在这个仓库里的具体建议

这个工作区是典型的 ROS2 多包仓库，最容易上下文爆炸的点是“全仓搜索 + build/install/log 噪音 + 视觉/导航/行为树混聊”。

建议你固定执行下面几条：

- 视觉问题只限定在 `src/sp_vision25`
- 行为树问题只限定在 `src/pb2025_sentry_behavior`
- 导航问题只限定在 `src/pb2025_sentry_nav`
- 只有消息定义或启动链变更时，才允许跨包
- 除非必要，不要把 `build/ install/ log/ logs/` 里的内容放进对话

## 8. 仓库内已经为你加好的辅助设施

- `AGENTS.md`: 给 Codex 的仓库级规则
- `.ignore`: 降低搜索噪音
- `patterns/`: 任务模板
- `scripts/codex_brief.sh`: 生成精简上下文摘要

## 9. 你的标准操作流程

### 轻量流程

1. 先确定这轮只做一个目标
2. 运行 `scripts/codex_brief.sh <package>` 看摘要
3. 用 `patterns/` 里的模板发任务
4. 让 Codex 在包级范围内实现和验证
5. 结束时让它输出 handoff
6. 下一轮换目标就开新对话

### 强约束流程

适合复杂任务或你最近频繁撞 context：

1. 在任务开头明确写“只允许改哪些路径”
2. 明确写“不要全仓构建”
3. 明确写“先分析，再改，再验证”
4. 明确写“如果发现跨包影响，先停下并说明”

## 10. 一个够用的短提示词

以后你可以直接这样开场：

```md
按仓库里的 `AGENTS.md` 和 `docs/codex_workflow.md` 执行。
目标: ...
范围: ...
约束: ...
验证: ...
现象: ...
```

这已经足够，比长篇背景更省 token，也更稳。
