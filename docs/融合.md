## 1. 文档目的

本文档用于详细说明这次“`sp_vision25` 视觉系统接入 `sentry` 导航/行为树决策链”的工程改动，重点回答下面几个问题：

1. 我是如何接入 ROS 接口的。
2. 为什么单独新建 `sp_msgs`，而不是把接口塞进 `pb_rm_interfaces`。
3. 哪些文件负责发布、哪些文件负责订阅、哪些文件负责行为树决策。
4. 视觉、自瞄、导航、行为树之间的话题和消息是怎么流动的。
5. 这次具体改了哪些文件、每个文件承担什么作用。
6. 当前这套“第一代 Artisan 导航融合系统”已经做到什么程度，后续还可以怎么继续扩展。

本文档默认你已经熟悉当前工程的大体结构，但不要求你记得每个包的入口文件。

## 2. 这次融合的总体思路

这次没有去复活旧的 `standard_robot_pp_ros2/2025_robot_decision.cpp`，而是完全沿用你当前已经在使用的**行为树主链**。

整体思路分成三层：

1. `sp_vision25` 负责把视觉侧的“目标是否有效、是否建议接管导航、云台该指向哪里、是否建议前往某个支援点”等信息，整理成统一的 ROS 消息并发布。
2. `pb2025_sentry_behavior` 负责订阅这个视觉融合消息，把它放到行为树黑板中，并在行为树里通过条件节点和动作节点决定：
   - 是否进入视觉接管分支
   - 是否控制云台
   - 是否让导航切到视觉建议目标点
3. `pb2025_sentry_bringup` 负责在 loopback / 仿真环境里伪造视觉消息，方便先测链路，不依赖真视觉程序。

也就是说，这次做的不是“把视觉代码硬塞进导航”，而是把它整理成：

- 一个独立消息接口层
- 一个视觉发布层
- 一个行为树消费层
- 一个测试/仿真层

这样做的好处是：

1. 调试简单，`ros2 topic echo` 就能看清楚视觉到底给了什么。
2. 行为树和视觉算法本体解耦，后面你改目标选择策略时不会牵一发动全身。
3. 以后要把视觉接管逻辑合并回 `rmul_2026.xml` 主树时，不需要重写接口。

## 3. 为什么新建 `sp_msgs`

你在中途明确提出，希望：

> 直接在导航中新建一个 `sp_msgs` 消息包，而不是融合在 `pb_rm_interfaces`

我最后按这个方案实现，原因如下：

1. `pb_rm_interfaces` 目前主要承载裁判系统、云台、底盘、串口侧的稳定接口。
2. 视觉融合消息属于“策略层 / 感知层接口”，变化会更频繁。
3. `sp_vision25` 原本代码里已经存在对 `sp_msgs` 的依赖痕迹，继续沿这个方向改，迁移成本最低。
4. 现场调试时，把视觉相关消息单独放进 `sp_msgs`，更容易一眼分辨“这是视觉侧话题，不是裁判系统或串口协议话题”。

因此这次我在：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/CMakeLists.txt](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/CMakeLists.txt)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/package.xml](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/package.xml)

新建了独立的 `sp_msgs` 接口包。

## 4. 新增消息设计

### 4.1 消息包位置

新增消息都放在：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg)

当前包含三类消息：

1. `EnemyStatusMsg`
2. `AutoaimTargetMsg`
3. `VisionTargetMsg`

### 4.2 `VisionTargetMsg` 的作用

这次真正用于“视觉融合导航”的核心消息是：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg/VisionTargetMsg.msg](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg/VisionTargetMsg.msg#L1)

它包含了这些关键字段：

1. `timestamp`
   作用：告诉行为树这条视觉结果是不是还新鲜。
2. `tracking`
   作用：视觉是否已经稳定追踪到目标。
3. `nav_hold`
   作用：视觉是否明确建议导航进入接管/压制状态。
4. `fire_permitted`
   作用：视觉当前是否认为满足发弹条件。
5. `target_id`
   作用：当前识别到的目标编号。
6. `suggested_goal_index`
   作用：视觉建议前往哪个导航点。当前阶段可以是 `-1`，表示还没给建议点。
7. `target_yaw / target_pitch`
   作用：行为树可以直接拿去发云台绝对角指令。
8. `target_position_gimbal`
   作用：目标在云台坐标系下的位置。
9. `target_position_map`
   作用：目标在地图坐标系下的位置。当前先预留，后续你做地图级融合时继续扩展。

这套字段设计的核心原则是：

1. 让行为树只读“它真正关心的结果”，而不是去理解视觉内部的 tracker/armor 细节。
2. 保持话题可观测，方便比赛现场直接 `echo`。
3. 为后续从“云台接管”扩展到“地图目标点融合”预留空间。

## 6. 行为树侧改动：`pb2025_sentry_behavior`

### 6.1 行为树服务端如何接入视觉话题

行为树服务端入口在：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/src/pb2025_sentry_behavior_server.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/src/pb2025_sentry_behavior_server.cpp#L88)

这里做了三件关键事情：

1. 新增参数：
   - `decision.topics.gimbal_cmd`
   - `decision.vision.topic`
   - `decision.vision.timeout_s`
2. 订阅视觉融合话题：
   - `decision_vision_topic_` 默认就是 `vision/target`
3. 把消息直接塞到黑板：
   - blackboard key = `sp_vision_target`

也就是说，行为树里任何节点只要读：

```text
{@sp_vision_target}
```

就能拿到最近一帧视觉融合消息。

### 6.2 为什么要放黑板而不是节点内部自己订阅

这是因为当前行为树框架已经在用统一的“服务端订阅 + 黑板共享”的模式处理：

1. 裁判系统消息
2. 导航反馈
3. 仿真模式切换

视觉消息继续沿同样模式接入，有几个好处：

1. 风格统一
2. 多个 BT 节点可以复用同一份视觉消息
3. 不会在每个节点里重复创建订阅器
4. 更适合 Groot / 行为树黑板调试

### 6.3 新增条件节点：`IsVisionTargetValid`

新增文件：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.hpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.hpp)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.cpp#L8)

这个条件节点负责：

1. 检查黑板里有没有视觉消息
2. 检查 `tracking == true`
3. 检查 `nav_hold == true`
4. 检查 `target_yaw / target_pitch` 是否有效
5. 检查消息是否过期，超时时间由 `decision.vision.timeout_s` 控制
6. 如果通过，就把以下结果输出到黑板：
   - `vision_gimbal_yaw`
   - `vision_gimbal_pitch`
   - `vision_target_id`
   - `vision_goal_index`

这一步的本质是：

- 把“视觉是否足以触发接管”从普通消息判断，变成了一个行为树条件节点

这样在 XML 里就可以非常自然地写：

```xml
<IsVisionTargetValid ... />
```

### 6.4 新增动作节点：`SelectVisionTargetPath`

新增文件：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/action/select_vision_target_path.hpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/action/select_vision_target_path.hpp)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/action/select_vision_target_path.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/action/select_vision_target_path.cpp#L6)

这个动作节点负责：

1. 从黑板读取视觉建议点 `vision_goal_index`
2. 如果视觉建议点有效，就用它生成 `nav_msgs/Path`
3. 如果视觉没有给出有效索引，就回退到 `decision_anchor_target_index`

这里这么设计，是为了兼容两个阶段：

1. 第一阶段：视觉已经能接管云台，但还没做地图目标点语义映射
2. 第二阶段：视觉进一步给出目标区域对应的导航支援点

现在这套实现已经为第二阶段预留好了位置。

## 7. 测试行为树：`vision_test.xml`

你原本提出的思路是：

### 7.1 `vision_override`

1. `IsVisionTargetValid`
2. `PublishGimbalAbsolute`
3. `SelectVisionTargetPath`
4. `SendNavThroughPoses`

也就是说，只要视觉消息满足接管条件：

1. 云台先指向视觉目标

这个树把视觉分支放在最前面：

1. 先发布默认转圈速度
2. 优先尝试 `vision_override`

   - `decision_simulation`
   - `decision_referee`

这个结构的好处是：

1. 不破坏你现在 `rmul_2026` 主链
2. 可以先独立测试视觉接管逻辑
3. 等你确认效果稳定，再决定是不是并回主树

```bash
source install/setup.bash
```

```bash
  use_rviz:=True \
  vision_tracking:=True \
  vision_nav_hold:=True \

```

### 8.5 启动后如何动态模拟视觉参数

如果你已经把 launch 跑起来了，那么最方便的方式不是重启，而是直接改假输入节点参数：

```bash
ros2 param set /fake_decision_sim_inputs vision_tracking true
ros2 param set /fake_decision_sim_inputs vision_nav_hold true
ros2 param set /fake_decision_sim_inputs vision_target_yaw 0.25
ros2 param set /fake_decision_sim_inputs vision_target_pitch -0.08
ros2 param set /fake_decision_sim_inputs vision_suggested_goal_index 2
```

常用参数建议这样理解：

1. `vision_tracking`
   作用：控制 `IsVisionTargetValid` 能不能先过第一道门。
2. `vision_nav_hold`
   作用：控制视觉是否明确请求接管导航。
3. `vision_target_yaw`
   作用：直接观察云台绝对角输出是否跟着变。
4. `vision_target_pitch`
   作用：和 `vision_target_yaw` 一起测试云台话题。
5. `vision_suggested_goal_index`
   作用：直接指定行为树应该尝试去哪个点。
6. `publish_vision_target`
   作用：设成 `false` 可以测试视觉消息突然中断后的回退行为。

例如：

```bash
ros2 param set /fake_decision_sim_inputs vision_suggested_goal_index 1
ros2 param set /fake_decision_sim_inputs vision_target_yaw 0.40
ros2 param set /fake_decision_sim_inputs vision_target_pitch -0.12
```

就表示：

1. 导航建议切到 1 号点
2. 云台看向偏左上方一点的位置

如果你想验证“视觉失效后是否回退到普通决策”，可以直接这样改：

```bash
ros2 param set /fake_decision_sim_inputs vision_tracking false
```

或者：

```bash
ros2 param set /fake_decision_sim_inputs vision_nav_hold false
```

如果你想模拟“视觉节点掉线 / 不再发布”，可以这样改：

```bash
ros2 param set /fake_decision_sim_inputs publish_vision_target false
```

然后等待超过 `decision.vision.timeout_s`，行为树就应该退出视觉分支。

### 8.6 如何更全面地测试这条融合链

推荐你按下面顺序测，这样最容易定位问题。

第一层：先看消息有没有真的发出来

```bash
ros2 topic echo /vision/target
```

你应该能看到：

1. `tracking`
2. `nav_hold`
3. `target_yaw`
4. `target_pitch`
5. `suggested_goal_index`

第二层：看云台话题有没有跟着视觉变化

```bash
ros2 topic echo /cmd_gimbal
```

然后修改：

```bash
ros2 param set /fake_decision_sim_inputs vision_target_yaw 0.30
ros2 param set /fake_decision_sim_inputs vision_target_pitch -0.05
```

如果 `/cmd_gimbal` 跟着变化，说明“视觉消息 -> BT 条件节点 -> 云台发布”已经通了。

第三层：看导航是不是会跟着建议点变化

推荐开 RViz，然后分别执行：

```bash
ros2 param set /fake_decision_sim_inputs vision_suggested_goal_index 1
ros2 param set /fake_decision_sim_inputs vision_suggested_goal_index 2
ros2 param set /fake_decision_sim_inputs vision_suggested_goal_index -1
```

这里的预期是：

1. `1` 时优先走 1 号点
2. `2` 时优先走 2 号点
3. `-1` 时回退到 `anchor_target_index`

第四层：测视觉分支失效后的回退

先让视觉分支正常接管，再执行：

```bash
ros2 param set /fake_decision_sim_inputs vision_tracking false
```

或者：

```bash
ros2 param set /fake_decision_sim_inputs publish_vision_target false
```

预期应该是：

1. 行为树不再停留在 `vision_override`
2. 系统重新回到 `decision_simulation` 或 `decision_referee`

第五层：绕过假参数节点，直接手发一条视觉消息

如果你想验证“只要有一条标准 `VisionTargetMsg`，行为树就能吃进去”，可以先关掉假视觉发布：

```bash
ros2 param set /fake_decision_sim_inputs publish_vision_target false
```

然后手发：

```bash
ros2 topic pub /vision/target sp_msgs/msg/VisionTargetMsg "{tracking: true, nav_hold: true, fire_permitted: false, target_id: 7, suggested_goal_index: 1, confidence: 1.0, target_distance: 3.0, target_yaw: 0.2, target_pitch: -0.05, target_position_gimbal: {x: 1.0, y: 0.0, z: 0.0}, target_position_map: {x: 0.0, y: 0.0, z: 0.0}}" -r 5
```

这个测试能把“假参数节点逻辑”和“行为树消费逻辑”拆开看。

第六层：验证真视觉程序自己的建议点逻辑

这里要特别说明：

1. `fake_decision_sim_inputs.py` 里的 `vision_suggested_goal_index` 是**直接注入**
2. 它验证的是“行为树是否正确消费视觉建议点”
3. 它**不验证** `sp_vision25` 自己的 `vision_fusion` 自动映射规则

如果你要验证：

- [/home/ats/ats_sentry_ws/sp_vision25/configs/sentry.yaml](/home/ats/ats_sentry_ws/sp_vision25/configs/sentry.yaml)

里那套 `vision_fusion.*` 参数是否真的生效，就需要启动真视觉发布端，再去看：

```bash
ros2 topic echo /vision/target
```

重点观察：

1. `suggested_goal_index`
2. `target_position_gimbal`
3. `target_distance`

是不是随着真实目标相对方位变化而自动变化。

## 10. 通信关系梳理

### 10.1 当前主通信话题

这次融合中最关键的话题如下：

1. `/vision/target`
   - 发布方：`sp_vision25` 或 `fake_decision_sim_inputs.py`
   - 订阅方：`pb2025_sentry_behavior_server`
   - 消息类型：`sp_msgs/msg/VisionTargetMsg`

2. `/cmd_gimbal`
   - 发布方：行为树中的 `PublishGimbalAbsolute`
   - 订阅方：底盘/云台控制侧
   - 消息类型：`pb_rm_interfaces/msg/GimbalCmd`

3. `/navigate_through_poses` action
   - 发送方：行为树中的 `SendNavThroughPoses`
   - 执行方：Nav2

4. `/decision/sim_mode`
   - 发布方：`fake_decision_sim_inputs.py`
   - 订阅方：行为树服务端
   - 作用：切换 simulation 模式

### 10.2 逻辑通信链

可以把整个链路抽象成：

```text
视觉识别/跟踪
  -> 生成目标摘要
  -> 发布 /vision/target
  -> 行为树判断是否接管
  -> 云台指向视觉目标
  -> 导航切到视觉支援点
```

进一步拆细一点：

```text
YOLO / Tracker / Aimer
  -> Decider::get_target_info()
  -> io::VisionTargetState
  -> sp_msgs/VisionTargetMsg
  -> blackboard(sp_vision_target)
  -> IsVisionTargetValid
  -> PublishGimbalAbsolute
  -> SelectVisionTargetPath
  -> SendNavThroughPoses
```
