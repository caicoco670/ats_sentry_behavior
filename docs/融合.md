本文档默认你已经熟悉当前工程的大体结构，但不要求你记得每个包的入口文件。

### 1.1 当前实现状态（2026-04-18）

如果你只关心“现在代码到底按什么方式工作”，先看这一节；后文如果与这一节冲突，以这一节为准。

如果你当前最需要的是“按最新版命令直接做 loopback 仿真调试”，建议先配合阅读：

- [视觉跟随仿真调试](/home/ats/ats_sentry_ws/docs/视觉跟随仿真调试.md)

1. `VisionTargetMsg.suggested_goal_index` 仍然保留在消息里，便于兼容旧版链路和调试观测，但**当前 `vision_test.xml` 已不再依赖它做路径规划**。
2. 当前视觉接管分支优先使用 `sp_vision` 给出的 `target_position_map`，通过 `SelectVisionFollowPath` 在敌方周围按 `attack_radius` 生成一个可行跟随点。
3. 默认跟随半径为 `2.0`，对应参数为：
   - `decision.vision.attack_radius`
   - `decision.vision.follow_occupied_threshold`
   - `decision.vision.follow_sample_count`
4. 如果 `target_position_map` 仍是零值、不是有限值，或者不在代价地图可用范围内，视觉路径生成会失败，行为树会回退到普通导航逻辑。
5. 这次已经定位并修复两个会导致“视觉接管但不规划”的关键问题：
   - `duration=0` 的 BT 发布节点以前会先返回 `RUNNING`，导致后续规划节点被饿死；现已改为立即返回 `SUCCESS`
   - `IsPathGoalReached` 以前会被旧路径残留的 `goal_succeeded` 短路，导致新视觉目标来了也被误判成“已经到达”；现已改成按当前 pose 与当前路径终点距离判断
6. loopback 已验证：
   - 启动后视觉分支可以到达 `SelectVisionFollowPath -> SendNavThroughPoses`
   - 运行中动态修改 `vision_target_position_map_x/y` 会触发新的视觉跟随目标
   - 关闭 `vision_tracking` 后会回退到普通 `decision_simulation` 分支

## 2. 这次融合的总体思路

这次没有去复活旧的 `standard_robot_pp_ros2/2025_robot_decision.cpp`，而是完全沿用你当前已经在使用的**行为树主链**。

整体思路分成三层：

1. `sp_vision25` 负责把视觉侧的“目标是否有效、是否建议接管导航、云台该指向哪里、目标在地图中的位置”等信息，整理成统一的 ROS 消息并发布。
2. `pb2025_sentry_behavior` 负责订阅这个视觉融合消息，把它放到行为树黑板中，并在行为树里通过条件节点和动作节点决定：
   - 是否进入视觉接管分支
   - 是否控制云台
   - 是否让导航切到敌方周围的视觉跟随点
3. `pb2025_sentry_bringup` 负责在 loopback / 仿真环境里伪造视觉消息，方便先测链路，不依赖真视觉程序。

也就是说，这次做的不是“把视觉代码硬塞进导航”，而是把它整理成：

- 一个独立消息接口层
- 一个视觉发布层
- 一个行为树消费层
- 一个测试/仿真层

这样做的好处是：

1. 调试简单，`ros2 topic echo` 就能看清楚视觉到底给了什么。
2. 行为树和视觉算法本体解耦，后面你改目标选择策略时不会牵一发动全身。
3. 以后要把视觉接管逻辑合并回 `rmul_2026.xml` 主树时，不需要重写接口。

## 3. 为什么新建 `sp_msgs`

你在中途明确提出，希望：

> 直接在导航中新建一个 `sp_msgs` 消息包，而不是融合在 `pb_rm_interfaces`

我最后按这个方案实现，原因如下：

1. `pb_rm_interfaces` 目前主要承载裁判系统、云台、底盘、串口侧的稳定接口。
2. 视觉融合消息属于“策略层 / 感知层接口”，变化会更频繁。
3. `sp_vision25` 原本代码里已经存在对 `sp_msgs` 的依赖痕迹，继续沿这个方向改，迁移成本最低。
4. 现场调试时，把视觉相关消息单独放进 `sp_msgs`，更容易一眼分辨“这是视觉侧话题，不是裁判系统或串口协议话题”。

因此这次我在：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/CMakeLists.txt](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/CMakeLists.txt)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/package.xml](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/package.xml)

新建了独立的 `sp_msgs` 接口包。

## 4. 新增消息设计

### 4.1 消息包位置

新增消息都放在：

当前包含三类消息：

1. `EnemyStatusMsg`
2. `AutoaimTargetMsg`
3. `VisionTargetMsg`

### 4.2 `VisionTargetMsg` 的作用

这次真正用于“视觉融合导航”的核心消息是：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg/VisionTargetMsg.msg](/home/ats/ats_sentry_ws/src/pb2025_sentry_nav/sp_msgs/msg/VisionTargetMsg.msg#L1)

它包含了这些关键字段：

1. `timestamp`
   作用：告诉行为树这条视觉结果是不是还新鲜。
2. `tracking`
   作用：视觉是否已经稳定追踪到目标。
3. `nav_hold`
   作用：视觉是否明确建议导航进入接管/压制状态。
4. `fire_permitted`
   作用：视觉当前是否认为满足发弹条件。
5. `target_id`
   作用：当前识别到的目标编号。
6. `suggested_goal_index`
   作用：保留给旧版“固定导航点建议”策略使用。当前 `vision_test.xml` 已不再依赖它做路径规划，但该字段仍可用于兼容和观测。
7. `target_yaw / target_pitch`
   作用：行为树可以直接拿去发云台绝对角指令。
8. `target_position_gimbal`
   作用：目标在云台坐标系下的位置。
9. `target_position_map`
   作用：目标在地图坐标系下的位置。当前视觉跟随路径生成的主输入就是它。

这套字段设计的核心原则是：

1. 让行为树只读“它真正关心的结果”，而不是去理解视觉内部的 tracker/armor 细节。
2. 保持话题可观测，方便比赛现场直接 `echo`。
3. 既支持云台接管，也支持基于地图坐标直接生成视觉跟随路径。

## 6. 行为树侧改动：`pb2025_sentry_behavior`

### 6.1 行为树服务端如何接入视觉话题

行为树服务端入口在：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/src/pb2025_sentry_behavior_server.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/src/pb2025_sentry_behavior_server.cpp#L88)

这里做了三件关键事情：

   - `decision.vision.topic`
   - `decision.vision.timeout_s`
2. 订阅视觉融合话题：
   - `decision_vision_topic_` 默认就是 `vision/target`
3. 把消息直接塞到黑板：
   - blackboard key = `sp_vision_target`

也就是说，行为树里任何节点只要读：

```text
{@sp_vision_target}
```

就能拿到最近一帧视觉融合消息。

### 6.2 为什么要放黑板而不是节点内部自己订阅

这是因为当前行为树框架已经在用统一的“服务端订阅 + 黑板共享”的模式处理：

1. 裁判系统消息
2. 导航反馈
3. 仿真模式切换

视觉消息继续沿同样模式接入，有几个好处：

1. 风格统一
2. 多个 BT 节点可以复用同一份视觉消息
3. 不会在每个节点里重复创建订阅器
4. 更适合 Groot / 行为树黑板调试

### 6.3 新增条件节点：`IsVisionTargetValid`

新增文件：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.hpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.hpp)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/condition/is_vision_target_valid.cpp#L8)

这个条件节点负责：

1. 检查黑板里有没有视觉消息
2. 检查 `tracking == true`
3. 检查 `nav_hold == true`
4. 检查 `target_yaw / target_pitch` 是否有效
5. 检查消息是否过期，超时时间由 `decision.vision.timeout_s` 控制
6. 如果通过，就把以下结果输出到黑板：
   - `vision_gimbal_yaw`
   - `vision_gimbal_pitch`
   - `vision_target_id`
   - `vision_goal_index`

这一步的本质是：

- 把“视觉是否足以触发接管”从普通消息判断，变成了一个行为树条件节点

这样在 XML 里就可以非常自然地写：

```xml

```

### 6.4 新增动作节点：`SelectVisionFollowPath`

新增文件：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/action/select_vision_follow_path.hpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/include/pb2025_sentry_behavior/plugins/action/select_vision_follow_path.hpp)
- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/action/select_vision_follow_path.cpp](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/plugins/action/select_vision_follow_path.cpp#L124)

这个动作节点负责：

1. 从黑板读取 `sp_vision_target.target_position_map`
2. 读取当前机器人 pose 和全局 costmap
3. 以敌方位置为圆心、`attack_radius` 为半径，在环上采样候选跟随点
4. 用全局代价地图过滤掉占用或越界的点
5. 输出一个朝向敌方的单点 `nav_msgs/Path`

这里这么设计，是为了兼容两个阶段：

1. 第一阶段：视觉已经能稳定给出敌方地图坐标，先实现“围绕敌方保持攻击半径”的直接跟随
2. 第二阶段：后续如果要引入更复杂的卡点、侧翼点或战术点，再在这个基础上扩展

当前实现还有两个重要约束：

1. 如果 `target_position_map` 仍是零值，节点会判为“视觉地图坐标尚未就绪”，返回 `FAILURE`
2. 当前默认假设 `target_position_map` 已与 global costmap 使用同一全局坐标系；如果未来真实视觉发的是别的 frame，还需要补 TF 转换

## 7. 测试行为树：`vision_test.xml`

你原本提出的思路是：

> 先参考 `rmul_2025.xml` 新建一个 `vision_test.xml` 进行测试

这次我实际做成了：

- [/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/behavior_trees/vision_test.xml](/home/ats/ats_sentry_ws/src/pb2025_sentry_behavior/behavior_trees/vision_test.xml#L1)

它里面包含两个树：

1. `vision_override`
2. `vision_test`

### 7.1 `vision_override`

这个子树的逻辑顺序是：

1. `IsVisionTargetValid`
2. `PublishGimbalAbsolute`
3. `SelectVisionFollowPath`
4. `IsPathGoalReached`
5. `SendNavThroughPoses`

也就是说，只要视觉消息满足接管条件：

1. 云台先指向视觉目标
2. 决策节点根据敌方地图位置生成一个跟随点
3. 如果当前还没到该点，就向 Nav2 下发新的 `NavigateThroughPoses`

### 7.2 `vision_test`

这个树把视觉分支放在最前面：

1. 先发布默认转圈速度
2. 优先尝试 `vision_override`
3. 如果视觉分支不成立，再退回原有：
   - `decision_simulation`
   - `decision_referee`

这个结构的好处是：

1. 不破坏你现在 `rmul_2026` 主链
2. 可以先独立测试视觉接管逻辑
3. 等你确认效果稳定，再决定是不是并回主树

1. 云台看向偏左上方一点的位置

   - 发布方：`fake_decision_sim_inputs.py`
   - 订阅方：行为树服务端
   - 作用：切换 simulation 模式

  -> blackboard(sp_vision_target)

### 13.2 地图级目标点融合仍是第一代
