# 视觉跟随仿真调试

## 1. 当前调试目标

当前 `loopback_vision_test.launch.py` 的调试目标已经切换为：

1. 视觉侧通过 `sp_msgs/msg/VisionTargetMsg` 发布 `/vision/target`
2. 行为树优先读取 `target_position_map`
3. 决策通过 `SelectVisionFollowPath` 以敌方位置为圆心、`attack_radius` 为半径生成一个跟随点
4. 如果视觉目标无效或超时，行为树回退到普通导航逻辑

需要特别注意：

1. `vision_suggested_goal_index` 仍然保留在消息和 fake 输入节点里，方便兼容和观测
2. 但当前 `vision_test.xml` 已经不再依赖它做路径规划
3. 现在真正驱动视觉跟随规划的是 `target_position_map`

## 3. 启动方式

建议先重新编译相关包：

```bash
source /opt/ros/humble/setup.bash
colcon build --packages-select behaviortree_ros2 pb2025_sentry_behavior pb2025_sentry_bringup --symlink-install
source install/setup.bash
```

推荐的 loopback 启动命令：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup loopback_vision_test.launch.py \
  use_rviz:=True \
  vision_tracking:=True \
  vision_nav_hold:=True \
  vision_target_yaw:=0.30 \
  vision_target_pitch:=-0.06 \
  vision_target_position_map_x:=5.0 \
  vision_target_position_map_y:=2.0 \
  vision_target_position_map_z:=0.0
```

这条命令默认会：

1. 加载 `sentry_behavior_vision_test.yaml`
2. 让行为树执行 `vision_test`
3. 启动 fake vision 发布 `/vision/target`
4. 让视觉分支优先接管导航

## 4. 运行中常用调试命令

### 4.1 查看 fake vision 是否在发地图坐标

```bash
ros2 topic echo --once /vision/target
```

应重点确认：

1. `tracking: true`
2. `nav_hold: true`
3. `target_position_map.x/y/z` 为非零有效值

### 4.2 在线切换视觉目标位置

```bash
ros2 param set /fake_decision_sim_inputs vision_target_position_map_x 2.5
ros2 param set /fake_decision_sim_inputs vision_target_position_map_y 4.2
ros2 param set /fake_decision_sim_inputs vision_target_position_map_z 0.0
```

预期：

1. 行为树日志出现新的 `Vision follow target=(2.50, 4.20) ...`
2. 如果当前未到新跟随点，会再次出现 `Send NavigateThroughPoses goal with 1 poses`

### 4.3 在线调整跟随半径

```bash
ros2 param set /pb2025_sentry_behavior_server decision.vision.attack_radius 2.5
```

常用参数：

- `decision.vision.attack_radius`
- `decision.vision.follow_occupied_threshold`
- `decision.vision.follow_sample_count`

### 4.4 验证视觉失效后的回退

```bash
ros2 param set /fake_decision_sim_inputs vision_tracking false
```

或者：

```bash
ros2 param set /fake_decision_sim_inputs publish_vision_target false
```

预期：

1. `IsVisionTargetValid` 失败
2. 行为树退出 `vision_override`
3. 回退到普通 `decision_simulation` 或 `decision_referee`

## 5. 推荐的验证顺序

### 5.1 先看 action 服务是否正常

```bash
ros2 action info /navigate_through_poses
```

应看到：

1. client 包含 `/pb2025_sentry_behavior_server`
2. server 包含 `/bt_navigator`

### 5.2 再看视觉消息

```bash
ros2 topic echo --once /vision/target
```

### 5.3 再看行为树关键日志

推荐观察这些日志特征：

1. `SelectVisionFollowPath`
2. `Vision follow target=(...) selected_goal=(...)`
3. `Send NavigateThroughPoses goal with 1 poses`
4. `IsVisionTargetValid`

如果你想打开更详细的 BT tick 日志，可以临时把：

- `src/pb2025_sentry_behavior/params/sentry_behavior_vision_test.yaml`

里的 `use_cout_logger` 改为 `true`，或者复制一份到 `/tmp` 后只改这个参数。

### 5.4 最后看回退逻辑

当视觉失效后，日志里应重新出现普通导航路径下发，例如：

- `Send NavigateThroughPoses goal with 2 poses`

这通常表示系统已经回到默认 patrol 分支。

## 6. 当前已经修复的两个关键卡点

### 6.1 0 时长发布节点饿死后续规划

历史问题：

1. `PublishSpinSpeed` / `PublishGimbalAbsolute` 继承自 `RosTopicPubStatefulActionNode`
2. 以前即使 `duration=0`，`onStart()` 也会先返回 `RUNNING`
3. 在当前 `ReactiveSequence` / `ReactiveFallback` 结构下，后面的 `SelectVisionFollowPath` 永远吃不到 tick

当前状态：

1. 这个问题已经修复
2. `duration<=0` 时会立即返回 `SUCCESS`

### 6.2 新视觉路径被旧成功状态误判为“已到达”

历史问题：

1. `IsPathGoalReached` 会优先读取旧的 `goal_succeeded`
2. 当视觉目标在线切换时，新路径还没真正比较距离，就可能被直接判成成功
3. 结果就是新目标来了，但不重新下发导航

当前状态：

1. 这个问题已经修复
2. 现在会按“当前位姿到当前路径终点的距离”判断是否已到达

## 7. 常见问题

### 7.1 修改了 `vision_suggested_goal_index` 但不重规划

这是当前版本的正常现象。

原因：

1. `vision_test.xml` 现在已经不靠 `vision_suggested_goal_index` 规划
2. 请改 `vision_target_position_map_x/y/z`
3. 或直接由真实 `sp_vision` 发布新的 `target_position_map`

### 7.2 `/vision/target` 在发，但视觉路径一直失败

优先检查：

1. `target_position_map` 是否为零值
2. `target_position_map` 是否为有限值
3. 目标点是否落在全局 costmap 合法范围内

当前实现里，零值地图坐标会被视为“尚未就绪”。

### 7.3 启动时报 `Address already in use`

说明上一次调试残留的行为树进程没有退出，常见于 Groot2 端口 `17777` 被占用。

建议：

1. 先清理旧的 `pb2025_sentry_behavior_server`
2. 再清理 `bt_navigator`、`fake_decision_sim_inputs`、`loopback_simulator`
3. 必要时执行 `ros2 daemon stop` 和 `ros2 daemon start`

### 7.4 视觉分支有日志，但没有真的导航

优先检查：

1. `/navigate_through_poses` action 是否存在
2. `/pb2025_sentry_behavior_server` 是否是 action client
3. 日志里是否出现 `Send NavigateThroughPoses goal with 1 poses`
4. Nav2 是否已经完成 lifecycle 激活

---

## 8. 建议的日常调试习惯

推荐把调试分成两层：

第一层：先在 loopback 中只验证行为树和导航是否吃得进 `target_position_map`

第二层：再把 fake vision 替换成真实 `sp_vision25`

这样可以把问题切开：

1. 如果 loopback 都不通，先看行为树 / Nav2 / bringup
2. 如果 loopback 通、真视觉不通，再看 `sp_vision25 -> /vision/target`

一句话总结：

- 当前最新版视觉仿真调试的主入口是 `target_position_map`
- 当前最新版视觉跟随规划的主参数是 `decision.vision.attack_radius`
- 当前最新版回退验证的主开关是 `vision_tracking` 和 `publish_vision_target`
