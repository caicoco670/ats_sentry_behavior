# MPC 小陀螺切板问题修复说明

## 1. 这次修改要解决什么问题

这次修改针对 `auto_aim_debug_mpc` 在实弹测试中的三个典型现象：

1. 从高装甲板切向低装甲板时，弹道持续偏高。
2. 预测点总感觉“靠前”，尤其在小陀螺切板时更明显。
3. 逆时针慢速小陀螺时，弹着点整体偏右。

这几个现象放在一起看，不像单一的 `yaw_offset / pitch_offset` 偏差，更像是：

1. `MPC` 选板逻辑不合理。
2. 参考轨迹在切板瞬间被上一帧状态污染。
3. 调试画面看不到真正关键的内部量，现场很难快速定位。

## 2. 修改了哪些文件

### 2.1 `tasks/auto_aim/planner/planner.hpp`

新增了：

1. `AimSelection` 结构体。
2. 若干 `debug_*` 调试字段。
3. `preview_aim_selection()` 接口。

目的：

1. 让 `Planner` 不只是输出最终 `Plan`，还能把“当前到底选中了哪块板”暴露出来。
2. 让 `auto_aim_debug_mpc` 可以复用 `Planner` 的真实选板逻辑，而不是自己再写一套“最近装甲板”的简化逻辑。

### 2.2 `tasks/auto_aim/planner/planner.cpp`

这是本次最核心的修改文件。

主要改动有四处：

#### A. 新增 `choose_aim_selection()`

原来 `Planner::aim()` 的做法是：

1. 遍历 `target.armor_xyza_list()`。
2. 永远选择 `xy` 平面距离最小的那块板。

这个逻辑在静止或弱机动目标下还能工作，但到了小陀螺场景会出现明显问题：

1. “最近板”未必是当前真正应该击打的板。
2. 切高低板时，最近板和真正入射板可能不是同一块。
3. 对逆时针/顺时针的旋向没有任何区分。

因此现在改成两套逻辑：

1. 非小陀螺：保留锁板思想，在可见窗口内尽量打同一块板，避免左右抖动。
2. 小陀螺：不再选最近板，而是根据 `w` 的符号以及 `comming_angle / leaving_angle`，优先选择“正在进入可射击窗口”的那块板。

这样做的直接意义是：

1. 逆时针和顺时针会有不同的切板方向。
2. `Planner` 终于真正用上了 `standard3.yaml` 里的小陀螺切板参数。
3. 可以明显减少“预测点总感觉靠前/靠后”的问题。

#### B. `plan(Target, bullet_speed)` 的飞行时间估计改掉了

原来这一步也是拿“最近装甲板”估计弹丸飞行时间。

问题在于：

1. 你后面真正 `aim()` 打的板，可能不是这块最近板。
2. 那么你前面算出来的飞行时间就已经偏了。
3. 一旦高低板之间 `z` 不同，`pitch` 会先天被带偏。

现在这一步先调用 `preview_aim_selection()`，用“当前真正准备击打的板”去估算飞行时间，再做预测。

这一步对“从高板切低板时一直偏高”尤其关键。

#### C. 去掉了 `get_trajectory()` 里的跨帧 `static last_yaw_vel`

原代码里：

1. `last_yaw_vel` 是 `static`。
2. 也就是说它会跨帧保存上一次规划周期的值。
3. 如果上一帧正好在切板，或者上一帧选的还是另一块板，这个速度信息就会污染当前帧。

这会造成：

1. 当前帧参考轨迹起始速度不干净。
2. 切板瞬间 `yaw_vel` 会被平白拉大或拉小。
3. 用户体感上就是“点位突然被往前推了一截”或者“总差半步”。

现在改为：

1. 每次进入 `get_trajectory()` 时重新初始化 `last_yaw_vel`。
2. 只在当前这一次预测地平线内部做平滑。
3. 不再让上一帧的切板状态污染这一帧。

#### D. 增加 `debug_delay_time`

`plan(std::optional<Target>, bullet_speed)` 里现在把实际使用的 `delay_time` 保存到了 `debug_delay_time`。

目的很简单：

1. 现场很多时候不是你不知道有 delay。
2. 而是你不知道当前这帧到底走的是高速 delay 还是低速 delay。
3. 现在可以直接在调试界面和 plotter 里看见了。

### 2.3 `src/auto_aim_debug_mpc.cpp`

这个文件的改动目标不是“改控制”，而是“把真正需要看的诊断量直接暴露出来”。

主要改动有五处：

#### A. 增加独立弹道调试面板 `Ballistic Debug`

这个面板会显示：

1. 理想弹道和当前下发弹道的侧视图。
2. 顶视图中的目标方向和实际射击方向。
3. 横向误差、纵向误差、总误差。
4. `yaw/pitch` 参考值与实际规划值之间的偏差。

它的意义是把“感觉打高/打右”变成可量化的毫米级误差，而不是靠主观判断。

#### B. 未来轨迹预览不再使用“最近装甲板”

原来的预览轨迹只是选未来时刻的最近板，因此它和 `Planner` 真正下发给云台的板号可能不一致。

这会导致一个非常致命的调试错觉：

1. 画面上看起来切板点在这里。
2. 实际 `Planner` 选的却是另一块板。
3. 于是你会误以为是弹道错了，实际上是选板错了。

现在未来轨迹预览改成调用 `debug_planner.preview_aim_selection()`，保证轨迹箭头和真实控制逻辑一致。

#### C. 左上角新增关键状态量

现在画面左上角会显示：

1. `armor_id`：当前 `Planner` 真正选中的装甲板编号。
2. `spin_gate`：当前是否进入小陀螺选板逻辑。
3. `delay`：当前实际使用的预测延迟。
4. `w`：目标角速度估计。
5. `h`：高低装甲板高度差估计。

现场判断时可以这样看：

1. 如果 `armor_id` 经常跳得不合理，先怀疑切板逻辑。
2. 如果 `armor_id` 合理但 `h` 飘得厉害，先怀疑高低板高度差估计。
3. 如果 `armor_id` 合理、`h` 也稳定，但 `delay` 明显偏大，再看预测超前问题。

#### D. plotter 新增内部量

现在 `plotter` 里多了：

1. `target_h`
2. `planner_selected_armor`
3. `planner_delay_ms`
4. `planner_spin_gate`

这样录曲线时，不只看 `yaw/pitch`，还能看到问题究竟是从状态估计、选板还是延迟来的。

#### E. 红色重投影目标点改为使用 `debug_planner.debug_xyza`

也就是说，红色目标点现在代表的是“Planner 真正准备打的板”，而不是某个独立计算出来的近似点。

## 3. 这次优化为什么能对应你的现场现象

### 3.1 从高板切低板时打高

这类问题往往有两种来源：

1. 选错板。
2. 高低板高度差 `h` 估计偏大。

原逻辑使用最近板估计飞行时间，又在 `aim()` 中重新选最近板，这很容易在切高低板时把预测时序和目标板搞乱。

现在的修复是：

1. 飞行时间估计使用真实准备打的板。
2. 真正瞄准点也使用同一套选板逻辑。
3. 调试界面直接显示 `h`。

这三步做完后，你能明确区分：

1. 是切板错。
2. 还是 `h` 估计错。
3. 还是纯 `pitch_offset` 问题。

### 3.2 预测点总感觉靠前

这一点原来最可疑的是两个地方：

1. `delay` 已经补偿了一次。
2. `get_trajectory()` 又把上一帧的 `static last_yaw_vel` 带进来了。

如果这两个效应同时存在，就很容易让参考轨迹在切板时显得“过于积极”。

这次代码层面做掉的是第二个问题，也就是把跨帧速度污染去掉。

### 3.3 逆时针慢速小陀螺总偏右

这个现象最符合“旋向处理不对”。

因为原来的 `Planner` 根本没有按 `w > 0 / w < 0` 区分进入窗口的方向，而是单纯选最近板。

现在小陀螺逻辑会：

1. 读取 `w` 的符号。
2. 根据 `comming_angle / leaving_angle` 选入射板。
3. 在未来轨迹可视化里也使用同一套逻辑。

所以这次修改最直接改善的，其实就是这一条。

## 4. 现在现场应该怎么用这套调试信息

建议你打靶时按下面顺序判断：

1. 先看左上角 `armor_id` 是否符合你肉眼看到的那块目标板。
2. 再看 `spin_gate` 是否在你认为进入小陀螺时正确打开。
3. 再看 `h` 是否在切高低板瞬间突然跳变过大。
4. 如果 `armor_id` 和 `h` 都正常，再看 `Ballistic Debug` 里的 `vertical miss` 和 `lateral miss`。
5. 如果 `vertical miss` 持续为正，说明当前弹道仍偏高；如果 `lateral miss` 在逆时针时总同号，说明旋向或 yaw 方向还有余量要调。

## 5. 这次没有改什么

本次没有直接改：

1. `standard3.yaml` 的参数值。
2. `target.cpp` 里的高低板状态模型。
3. `yaw_offset / pitch_offset` 本身。

原因是这次先把“控制逻辑”和“诊断信息”修正到可信状态。

只有先确定：

1. 现在打的是哪块板。
2. 当前 `h` 是多少。
3. 当前 delay 走的是哪条分支。

后面再去调 `yaml`，才不会把“模型问题”误判成“参数问题”。

## 7. 后续建议

如果你接下来继续实弹联调，优先观察这三项：

1. `armor_id`
2. `h`
3. `vertical miss`

因为这三项最能直接回答你现在最关心的问题：

1. 是不是高低板切错了。
2. 是不是高度差估计飘了。
3. 还是单纯 pitch 补偿还要再压一点。
